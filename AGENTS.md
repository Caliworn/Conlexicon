# AGENTS.md

本文件记录 Conlexicon 项目中长期有效的协作、实现和验收规则。阶段性计划、临时交接和下一步任务放在 `NEXT_IMPLEMENTATION_HANDOFF.md`。

## 工作流

- 开始工作前先检查工作树，确认已有改动归属：
  ```bash
  git status --short
  ```
- 开工前先判断用户请求类型和授权边界：
  - 诊断、解释、评估、设计讨论：优先只读分析，不主动改代码。
  - 实现、修复、整理、生成：可以按请求改动文件，并做相称验证。
  - Git 操作、删除、迁移、批量重写、外部发布：只有用户明确要求时才执行。
  - 如果一个合理假设会明显改变任务范围，先说明并请求确认。
- 默认不要回滚、覆盖或删除用户已有改动。除非用户明确要求，不要执行破坏性 Git 操作。
- 文件编辑优先使用 `apply_patch`。复杂文本处理优先使用 Git Bash、`rg`、Node 脚本或专用工具，尽量避免 PowerShell 复杂管道。
- 搜索文件或文本优先使用 `rg` / `rg --files`。
- Windows 环境优先使用 PATH 中的 Git Bash，即 `shell: "bash"`；只有 PATH 不可用时才退回完整路径 `C:\Program Files\Git\bin\bash.exe`。
- 不要把测试数据写入真实 `data/`。需要测试数据时，使用临时目录并通过 `CONLEXICON_DATA_DIR` 指向它。
- `data/` 是用户私有数据，禁止提交、覆盖或作为测试夹具使用。

## 本地服务和测试端口

- 默认开发服务端口是 `4173`。
- 启动测试服务前，先检查端口是否已有服务响应；如果已有可用服务，优先复用，不要重复启动。
- 不要假设测试服务已经被成功关闭。即使命令返回或会话结束，也要用端口探测确认。
- 如需启动临时服务，优先使用临时数据目录和明确的环境变量，避免写入真实 `data/`。
- 如果测试进程没有可靠关闭，后续回归测试应先复用既有端口或提示用户手动处理；不要反复启动新的后台服务堆积进程。

## 基本检查

常规代码改动后至少执行：

```bash
node --check app.js
node --check server.js
```

如改动 CSS、布局、交互或文档结构，视风险补充浏览器检查、关键宽度检查或 `git diff --check`。

## Changelog 和 README

- 未提交的新变更先写入 `CHANGELOG.md` 的 `New` 小节。
- `New` 下尽量按“新增 / 改进 / 修复 / 性能 / 安全与兼容性”等小标题归类；没有必要时可省略空标题。
- 每次提交前，将当前 `New` 下已有内容转化为提交当天日期。
- 如果同一自然日期内已经存在本次提交对应的日期段，则为新的提交日期段编号，例如：
  - `## 2026-07-02`
  - `## 2026-07-02 (2)`
  - `## 2026-07-02 (3)`
- 日期转换后，保留空的 `## New`，之后任何新增改动继续写入 `New` 下。
- 重大功能更新、用户可见行为变化、运行方式变化、数据存储变化或快捷键变化，需要检查并更新 `README.md`。

## 数据不变量

- 词条、语料块、语料层、语料单元等实体 ID 必须保持唯一；保存和导入都要验证。
- 语料单元只能有一个父级：孤立、直属语料块，或属于某个语料层。
- 语料层在语料块内必须有显式顺序；语料块直属单元和语料层内单元也必须有显式顺序。
- 导入相同词典 ID 时必须保留“导入并覆盖/取消”确认，不能静默覆盖已有词典。

## 编辑与导航不变量

- 词条编辑分为完整编辑和局部编辑；离开当前词条或切换页面前，必须按当前词典设置分别处理完整编辑和局部编辑：直接保存、直接放弃，或弹出“保存 / 放弃更改 / 取消”三选项。
- 用户在三选项弹窗中选择取消时，必须中断当前切换、跳转、新建或其他会离开当前编辑上下文的操作。
- 词典管理、其他设置、自动 IPA、自动形态学等配置页存在未保存表单时，离开前必须复用应用内“保存 / 放弃更改 / 取消”确认流程。
- 语言文档和语料库受各自自动保存开关控制：自动保存开启时，编辑内容应持续保存且不退出编辑状态；自动保存关闭时，离开前必须走同样的保存确认流程。
- 自动保存不得退出编辑状态、清空草稿、重置选择、改变滚动位置或破坏撤销历史。
- 新增的导航入口都必须接入相同的未保存检查，包括模块切换、词条切换、抽屉内操作、词汇网络跳转、高级筛选跳转、右键菜单操作和快捷键触发的新建词条。
- 删除、批量重写、标签自动排序等风险操作使用独立的确认弹窗；它们不应被误归入“保存 / 放弃 / 取消”离开确认流程。
- 不使用浏览器原生 `confirm` / `alert`，统一复用应用内弹窗。

## 持久化边界

- 词典内容和词典级设置写入对应词典文件。
- 全局界面偏好写入 `data/index.json`，目前包括当前词典、界面语言和界面主题。
- 页面滚动位置、数据分析页码、临时面板展开状态、抽屉开闭状态等进程 UI 状态不写入词典文件。
- 数据模型升级时，优先保证旧版本 JSON 能导入并转换成新格式；除非确有必要，不要为了旧前端数据形状额外维护临时兼容层。前端因新数据模型出问题时，应修前端而不是绕回旧模型。
- 旧版本 JSON 字段迁移集中放在 `lib/legacy-dictionary-migration.js`；核心规范化模块只处理当前数据形状的默认值、ID 补齐和当前语义校验，不继续新增旧字段兼容逻辑。
- SQLite 是正式运行期主存储。JSON repository 只保留为 legacy/debug/迁移参考；后续新增功能、schema 升级、读取优化、保存返回值优化和前端新数据模型默认不再同步更新 JSON repository。只有修复旧 JSON 导入/转换、基础读取、显式回滚或 legacy 导出相关问题时，才应修改 JSON repository。
- 任何对象 SQL 化前都必须先判断必要性：结构性主数据可以 SQL 化；语言源码优先文本化；低频复杂配置可 blob 化；派生结果只作索引或缓存；没有明确查询、局部保存、一致性或性能收益的对象不要拆表。

## API 契约

- 前后端本地 HTTP API 的长期契约记录在 `API_CONTRACT.md`。
- 修改 API 路径、请求体、响应体、错误码、保存范围或兼容层边界时，必须同步检查并更新 `API_CONTRACT.md`。
- 普通运行期保存优先使用细粒度或模块级端点；完整词典快照端点只作为兼容层、完整导入/导出和低频管理入口。
- 前端应依赖结构化错误码做本地化提示，不应依赖英文错误文本。

## 性能与渲染原则

- 大列表和重型分析不得在非必要时全量渲染或全量计算。
- 当前词条列表、词根衍生词和语料列表的虚拟滚动机制不得在没有替代方案时移除。
- 数据分析未打开时不应生成完整分析报告。
- 如果后续改用分页、索引查询或服务端窗口化，可以替代现有虚拟滚动，但必须保持同等或更好的大数据集可用性。

## UI 验收检查项

新增或重构 UI 后，至少检查：

- 中文和英文界面文案。
- 浅色和深色主题。
- 长词典名、长工具名、长标签和长高级筛选名。
- 键盘操作和应用内 tooltip / 弹窗层级。
- 320、480、768、1024、1440px 等关键宽度。
